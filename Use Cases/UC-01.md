# Conference Management System — Use Case

## Use Case 1: Register Account

**Goal in Context**: Allow a public user to create an account to access CMS features  
**Scope**: Conference Management System  
**Level**: User Goal  
**Primary Actor**: Public User  
**Secondary Actors**: Email Service  
**Trigger**: User selects “Register”  

## Success End Condition
* A new user account is created in `pending` state and is activated only after successful email
  confirmation.

## Failed End Condition
* No account is created

## Preconditions
* User is not logged in

## Main Success Scenario
1. Public user selects the registration option
2. System displays registration form
3. User enters required details
4. User submits the form
5. System validates the information
6. System creates the account in `pending` state
7. System sends confirmation email
8. User opens the confirmation link
9. System validates the token and activates the account

## Extensions
* **5a**: Invalid or missing information  
  * **5a1**: System displays validation errors and does not create an account
* **5b**: Email already registered
  * **5b1**: System rejects registration, displays "Email already registered", and provides
    login/reset-password guidance
* **5c**: Too many attempts for the same email within 10 minutes
  * **5c1**: System temporarily blocks registration and returns retry guidance
* **5d**: Password policy fails
  * **5d1**: System displays password policy errors and does not create an account
* **7a**: Initial confirmation email delivery fails
  * **7a1**: System keeps account pending, queues retry, and informs user that email delivery is
    pending
* **8a**: Confirmation token is invalid, expired, or already consumed
  * **8a1**: System does not activate the account and informs user confirmation failed
